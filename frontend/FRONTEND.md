# 💻 SigmaTrack — Документация Frontend Client

Клиентская часть системы управления проектами и задачами (SigmaTrack) разработана на фреймворке **Nuxt 4** с использованием экосистемы **Vue 3 (Composition API)** и полной статической типизацией на **TypeScript**.

Интерфейс спроектирован как высокопроизводительное SPA/SSR приложение, полностью адаптированное для работы в Docker-контейнерах за Reverse Proxy.

---

## 🛠️ Архитектура и Особенности Конфигурации

Конфигурация приложения (`nuxt.config.ts`) оптимизирована под стандарты экосистемы Nuxt 4:

* **Nuxt 4 v4-Future Ready:** Включена опция `future.compatibilityVersion: 4`, активирующая современную структуру папок (`app/`), строгое разделение слоев и улучшенные дефолтные настройки производительности компиляции.
* **Оптимизация Бандла Иконок:** Модуль `@nuxt/icon` настроен в гибридном режиме с `serverBundle: 'local'` и клиентским сканированием кода (`clientBundle: { scan: true }`). Это позволяет автоматически упаковывать в итоговый JS-пакет только те иконки из сетов, которые реально импортированы в компоненты, исключая раздувание билда.
* **Управление состоянием (State Management):** **Pinia** используется для распределенного менеджмента данных. Сторы изолированы по зонам ответственности (аутентификация, текущий активный проект, список задач и менеджер push-уведомлений).

## 🌐 Интернационализация (i18n)

Реализована частичная поддержка мультиязычности (RU/EN) на базе модуля `@nuxtjs/i18n`:

* **Стратегия маршрутизации (`no_prefix`):** Локализация работает без изменения URL-адресов. Текущий язык пользователя сохраняется в cookie, обеспечивая монолитный и чистый UX.
* **Архитектура локалей:** Настройки вынесены в `~/i18n/i18n.config.ts`. Файлы переводов разбиты на изолированные модули.

## ⚙️ Переменные окружения и Настройка `.env`

Для связи с API бэкенда используется механизм `runtimeConfig`. При локальной разработке браузер выполняет запросы асинхронно, ориентируясь на адрес, переданный в `NUXT_PUBLIC_API_BASE`.

### Настройка для локальной разработки (Без Docker):

1. В папке `frontend/` скопируйте шаблон конфигурации:
```bash
cp .env.example .env

```

2. Откройте созданный `.env` и укажите прямой адрес локального API бэкенда (ASP.NET Core):
```bash
NUXT_PUBLIC_API_BASE=http://localhost:5069

```

### Настройка для Production / Docker Compose (За Reverse Proxy):

Когда вся система разворачивается через `docker-compose` с Nginx в качестве единой точки входа (слушающей, например, порт 80), запросы от браузера должны идти на единый домен/хост без указания порта бэкенда:

```bash
# Для работы на локальном сервере через Nginx
NUXT_PUBLIC_API_BASE=http://localhost

# Для развертывания на удаленном сервере (VPS)
# NUXT_PUBLIC_API_BASE=[http://195.201.12.34](http://195.201.12.34)
# NUXT_PUBLIC_API_BASE=[https://sigmatrack.ru](https://sigmatrack.ru)

```

*При такой схеме Nginx перехватывает запрос браузера к `http://localhost/api/...` и проксирует его во внутреннюю сеть Docker на `http://backend:5069`, полностью избавляя приложение от необходимости настройки CORS.*

## 🚀 Скрипты запуска (CLI)

Управление проектом осуществляется через стандартный менеджер пакетов **npm**.

Установка всех зависимостей проекта:

```bash
npm install
```

Запуск локального сервера разработки (с hot-reload):

```bash
npm run dev
```

Сборка оптимизированного Production-бандла:

```bash
npm run build
```

Локальный запуск и проверка собранного Production-приложения (SSR Node.js сервер):

```bash
npm run preview
```

## 🐳 Контейнеризация (Docker)

В корне папки `frontend/` находится изолированный `Dockerfile` для сборки образа фронтенд-приложения. Сборка оптимизирована и проходит в два этапа (**Multi-stage build**):
1. **Builder Stage:** Использование базового образа `node:24-alpine`, установка зависимостей через стандартный `npm` для обеспечения максимальной стабильности путей в изолированной файловой системе Docker и компиляция готового Production-билда Nuxt (`npm run build`).
2. **Runner Stage:** Создание чистого финального образа, куда копируется только готовый скомпилированный результат `.output`. Все исходники и служебные файлы разработки отсекаются.

Контейнер работает в Production-режиме, слушает порт `3000` на хосте `0.0.0.0` и запускает легковесный автономный сервер:
```bash
CMD ["node", ".output/server/index.mjs"]

```

> 💡 Оркестрация контейнера фронтенда совместно с бэкендом и Nginx описана в общем файле `docker-compose.yml` в корне репозитория. См. [Главный README.md](../README.md).
